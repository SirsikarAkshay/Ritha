# Ritha — Flutter mobile app

The iOS / Android client for Ritha. Mirrors the web app's screens and talks to
the same Django backend.

## Identifiers

| Platform | Identifier            | Where it's set |
|----------|-----------------------|----------------|
| Android  | `com.ritha.ritha_mobile` | `android/app/build.gradle.kts` (`applicationId` + `namespace`) |
| iOS      | `com.ritha.rithaMobile`  | `ios/Runner.xcodeproj/project.pbxproj` (`PRODUCT_BUNDLE_IDENTIFIER`) |

The platforms use different ID conventions (snake_case for the Java package on
Android, camelCase for the iOS bundle ID) — that's expected.

## Firebase setup

The Firebase config files —

- `android/app/google-services.json`
- `ios/Runner/GoogleService-Info.plist`

— are committed to this private repo so collaborators and CI can build without
out-of-band setup. They contain the Firebase project ID and OAuth client IDs;
neither is a secret (clients ship them in the app binary anyway). Real access
control is enforced server-side via Firebase Security Rules.

`lib/firebase_options.dart` is auto-generated by `flutterfire configure` and
gets committed alongside the two files above whenever the Firebase project
changes.

The repo currently has the arokah-keyed files removed and `firebase_options.dart`
stubbed out so misconfigured builds fail loudly rather than silently talking to
a stale project. To attach a fresh Firebase project:

1. **Create the Firebase project** at [console.firebase.google.com](https://console.firebase.google.com).
   Pick the project ID carefully; it's permanent.
2. **Add an Android app** — package name = `com.ritha.ritha_mobile`.
   Download the resulting `google-services.json` into `android/app/`.
3. **Add an iOS app** — bundle ID = `com.ritha.rithaMobile`.
   Download the resulting `GoogleService-Info.plist` into `ios/Runner/`.
   Then open `ios/Runner.xcworkspace` in Xcode and drag the plist into the
   Runner target so it's registered in `project.pbxproj`. Just dropping it
   on disk is not enough — the iOS build won't bundle it.
4. **Generate `firebase_options.dart`**:

   ```bash
   dart pub global activate flutterfire_cli
   cd mobile_flutter
   flutterfire configure --project=<your-firebase-project-id>
   ```

   Select `android`, `ios`, and `web` when prompted. The CLI overwrites
   `lib/firebase_options.dart` with values from your new project.

If `firebase_options.dart` hasn't been regenerated,
`DefaultFirebaseOptions.currentPlatform` throws a `StateError` with
instructions — the app won't silently talk to a stale project.

Once `flutterfire configure` finishes, commit the three regenerated artifacts
together:

- `android/app/google-services.json`
- `ios/Runner/GoogleService-Info.plist`
- `lib/firebase_options.dart`

## Building

```bash
flutter pub get

# Android (release APK for sideloading)
flutter build apk --release --dart-define=API_BASE_URL=https://api.example.com/api

# iOS (release IPA for TestFlight)
flutter build ipa --release --dart-define=API_BASE_URL=https://api.example.com/api
```

`API_BASE_URL` is read at build time so the same source produces dev / staging
/ production binaries without committing URLs.

## Testing

```bash
flutter analyze
flutter test
```
